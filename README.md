# Agent Workflow Starter

A reusable, project-agnostic **multi-agent workflow** for coordinating AI coding agents (Claude or Codex CLIs) on a single project. It ships a clean three-role core — **Planner**, **Worker**, **Reviewer** — backed by a small local task-board server, a JSON board, and a read-only HTML viewer.

Copy this repo into a new project, adjust a few config values and role files, and start coordinating agents.

## How it works

- **Planner** turns the owner's requirements into deduplicated `todo` tasks on the board.
- **Worker** claims one task at a time, implements it, and moves it to `review`.
- **Reviewer** reviews completed work and either approves it to `done` or opens a follow-up `todo`.

Each agent runs in its own chat and never switches roles. The board (`task-board/board.json`) is the single source of truth and the coordination lock. A local Python server exposes a REST API so claims and status moves are atomic; agents fall back to editing the JSON directly when the server is not running.

Tasks flow through six columns: `todo → claimed → review → reviewing → done → archived`. The HTML viewer collapses these into three visual columns (To Do, Review, Done) and shows each active agent as a colored chip (background = model, border = role).

See `workflow/workflow-overview.md` for the full model and `workflow/api-guide.md` for the API.

## Requirements

- **Python 3.10+** (standard library only — no third-party packages).
- A coding-agent CLI: **Claude** (`claude`) and/or **Codex** (`codex`). Both are supported; configure the command names in `task-board/config.json`.

## Quick start

```bash
# 1. Start the task-board backend (serves API + viewer)
cd <path-to>/agent-workflow-starter
python task-board/server.py
# → "Project Task Board: serving at http://127.0.0.1:4177/viewer.html"

# 2. Open the viewer in a browser
#    http://127.0.0.1:4177/viewer.html

# 3. In a new agent chat, load a role with its start phrase:
#    "load as planner"   → create tasks
#    "load as worker"    → implement tasks
#    "load as reviewer"  → review completed work
```

On Windows PowerShell, use `python task-board\server.py`.

## Adapt to your project

1. **Edit `task-board/config.json`:**
   - `projectName`, `boardTitle` — shown in the viewer.
   - `owner` / `ownerLabel` — who requests and accepts work (replaces a hardcoded name).
   - `devServerUrl` — base URL used in inspection-target examples (e.g. your local app).
   - `host` / `port` — where the server listens.
   - `spawn.claudeCommand` / `spawn.codexCommand` — the CLI command names on your machine.
2. **Tailor the role files** (`roles/*.md`): fill in the project-specific "do not edit" directories in `planner.md`/`reviewer.md`, and specialize the Reviewer's two review lenses (quality + correctness) for your domain — search each file for the `> Adapt for your project` notes.
3. **(Optional) Edit `agent-color-schema.json`**: change `personalNamePool` to names you like.
4. **(Optional) Add upstream roles**: if you need research/audit agents that feed the Planner, follow the "Extending the workflow" section in `workflow/workflow-overview.md` (upstream agents write handoff files; only the Planner creates tasks).
5. **Create the Notion page** (optional): paste `notion-page.md` into Notion as a human-readable master copy to share with your team.

## Notes

- Runtime artifacts (`*.log`, `active-agents.json`, `agent-dispatch-settings.json`, `spawned-agent-logs/`, `*.json.tmp`) are git-ignored; they are regenerated by the server.
- The board is written atomically (temp file + replace), so a crash mid-write won't corrupt `board.json`.
